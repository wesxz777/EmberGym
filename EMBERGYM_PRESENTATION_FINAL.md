MEMBERS connect to MEMBERSHIPS and BOOKINGS. CLASSES are linked to TRAINERS and have multiple BOOKINGS. PAYMENTS track all transactions. Every connection has a relationship, and these relationships ensure data integrity. If you try to delete a member with active bookings, the system prevents it. This is what makes the data reliable."

### Slide 11: Data Flow - How Information Moves
```
╔═════════════════════════════════════════════════════════════╗
║   DATA FLOW DIAGRAM - BOOKING TRANSACTION                   ║
║   From Member Click to Confirmed Booking                    ║
╚═════════════════════════════════════════════════════════════╝

USER ACTION
│
├─ Member clicks "Book Class"
│
├─ Frontend sends REST request
│  (class_id, member_id, date, time)
│
├─ HTTPS encrypted transmission
│
├─ Backend receives request
│
├─ Validates data (has member booked before? Is class full?)
│
├─ Checks seat availability in CLASSES table
│
├─ Inserts record into BOOKINGS table
│
├─ Inserts record into PAYMENTS table (if applicable)
│
├─ Queries to get member email & SMS
│
├─ Sends confirmation email via email service
│
├─ Updates frontend with success message
│
└─ Member sees confirmation on screen

RESULT: Complete transaction in ~500ms
STATUS: ✅ Booking saved to database
STATUS: ✅ Payment recorded
STATUS: ✅ Confirmation sent
```

**Speaker (Member 3):**
> "When a member books a class, here's what happens behind the scenes. First, their click triggers a secure request to our backend. We validate everything—does this member exist? Is the class still available? Are there enough seats? We then save the booking to the database and process any payment if needed. We send confirmation via email. The entire process takes about half a second. All of this is automated and secure."

**Design Tips:**
- Sequential flow from top to bottom
- Show each step clearly
- Include data validation points
- Show end result with checkmarks
- Timing information included
- Use arrows to show flow direction
